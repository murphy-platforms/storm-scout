Executive Summary

A security assessment of the Storm Scout weather advisory dashboard identified 1 critical, 1 high, and 3 medium severity vulnerabilities. The most severe finding is an unauthenticated API endpoint that allows anyone on the internet to modify site operational status, potentially causing safety risks or operational disruption.

Key Findings:

  • Critical: Write endpoints lack authentication (immediate remediation required)
  • High: Cross-site scripting (XSS) vulnerabilities in weather data display
  • Medium: Missing security headers, SRI, and rate limiting configuration issues

Positive Findings: SQL injection protections are properly implemented. CORS is correctly restricted. Error handling does not leak sensitive information in production.

System Overview

Storm Scout is a weather advisory monitoring dashboard used by IMT and Operations teams to track NOAA weather alerts affecting 219 US testing centers. The system automatically ingests weather data every 15 minutes and provides operational status recommendations.

Component Technology Notes
Frontend Bootstrap 5.3, Vanilla JavaScript Two UI variants (Classic & Beta)

Database MariaDB 11.4.9 mysql2 driver with parameterized queries

External Data NOAA Weather API Read-only, HTTPS, 15-minute refresh

Detailed Findings

BE-01: Unauthenticated Write Endpoints
CRITICAL

OWASP Category: A01:2021 - Broken Access Control

Description: The operational status API endpoints allow any unauthenticated user to modify site operational status. This includes endpoints to change individual site status and bulk update multiple sites simultaneously.

Affected Endpoints:

  • POST /api/operational-status/sites/:id
  • POST /api/operational-status/bulk-update

Evidence: Manual testing confirmed the endpoint accepts POST requests without authentication. The route handler in operational-status.js has no authentication middleware.

Business Impact:

  • Safety Risk: Attacker could mark dangerous sites as "open" during severe weather
  • Operational Disruption: Attacker could mark all sites as "closed" unnecessarily
  • Impersonation: The decision_by field accepts any string, allowing audit trail manipulation

Proof of Concept:

curl -X POST https://your-domain.example.com/api/operational-status/sites/1 \ -H "Content-Type: application/json" \ -d '{"operational_status":"closed", "decision_by":"Attacker", "decision_reason":"Malicious"}'

Recommendation: Implement authentication immediately. Options include HTTP Basic Auth with environment-configured credentials, API key authentication, or integration with existing identity provider.

FE-01: Cross-Site Scripting (XSS) via NOAA Data
HIGH

OWASP Category: A03:2021 - Injection

Description: NOAA weather advisory data is rendered into the DOM using innerHTML without proper HTML escaping. If NOAA data were compromised or the database manipulated, malicious JavaScript could execute in users' browsers.

Evidence: Code review identified 20+ files using innerHTML with unescaped external data, including:

  • advisories.html - line 267: Advisory type, severity rendered directly
  • site-detail.html - line 362: Full advisory details rendered
  • index.html - lines 472, 534: Dashboard summaries

Business Impact: If exploited, attackers could steal session cookies, redirect users to phishing sites, or deface the dashboard. While NOAA data is generally trusted, defense-in-depth requires proper output encoding.

Recommendation: Create an escapeHtml() utility function and apply it to all dynamic content before DOM insertion. Alternatively, use textContent instead of innerHTML where HTML formatting is not required.

BE-06: Missing Security Headers
MEDIUM

OWASP Category: A05:2021 - Security Misconfiguration

Description: The application does not implement security headers typically provided by the helmet middleware. Missing headers include:

  • X-Frame-Options - Clickjacking protection
  • X-Content-Type-Options - MIME sniffing prevention
  • Content-Security-Policy - XSS mitigation
  • Strict-Transport-Security - HTTPS enforcement

Evidence: HTTP response header inspection shows only basic headers. The helmet package is not present in package.json.

Recommendation: Install helmet package and configure with appropriate Content Security Policy directives.

FE-04: Missing Subresource Integrity (SRI)
MEDIUM

OWASP Category: A08:2021 - Software and Data Integrity Failures

Description: Bootstrap CSS and JavaScript are loaded from jsDelivr CDN without Subresource Integrity (SRI) attributes. If the CDN were compromised, malicious code could be injected.

Evidence: Review of HTML files shows CDN resources loaded without integrity or crossorigin attributes.

Recommendation: Add SRI hash attributes to all CDN-loaded resources. Bootstrap provides these hashes in their official documentation.

BE-02: Rate Limiting Bypass Risk
MEDIUM

OWASP Category: A04:2021 - Insecure Design

Description: Rate limiting is implemented using express-rate-limit, but the application does not configure trust proxy setting. Behind the LiteSpeed proxy, attackers may bypass rate limiting by manipulating the X-Forwarded-For header.

Evidence: Code review of app.js shows no app.set('trust proxy', 1) configuration.

Recommendation: Add app.set('trust proxy', 1) to properly trust the first proxy and ensure rate limiting uses the correct client IP.

Verified Safe Controls

DB-01: SQL Injection Protection
✓ SAFE

All database queries use parameterized statements with the mysql2 driver. No string concatenation was found in query construction. This effectively mitigates SQL injection risk.

BE-05: CORS Configuration
✓ SAFE

HTTP response headers confirm CORS is properly restricted to https://your-domain.example.com. Cross-origin requests from other domains are blocked.
